# Agent Harness Rules

## 목적

이 폴더는 AI 에이전트가 이 저장소에서 작업할 때 필요한 최소 하네스 규칙을 정의한다.
운영 코드가 아니라 작업 지침이며, MVP에서는 문서 기반 규칙으로 시작한다.

이 하네스는 에이전트에게 어떤 맥락을 어떤 우선순위로 주입하고, 어떤 도구와 검증 절차를 허용하며,
작업 후 무엇을 추적할지에 집중한다.

## 문서 구성

- `CONTEXT_PACK.md`: Context Pack 구성 단위, 우선순위, 크기 제한, 추적 기준
- `AGENT_TASK_TEMPLATE.md`: 에이전트 작업 요청 입력 템플릿
- `DESIGN_CONTEXT.md`: 설계 문서에서 추출한 필수 제품/아키텍처 맥락
- `EXECUTION_POLICY.md`: Agent 실행 권한, 가드레일, 승인 기준
- `TOOL_REGISTRY.md`: 도구 목록, 권한 매트릭스, 실패 응답 기준
- `VERIFICATION_GATE.md`: 테스트, 린터, CI 기반 검증 게이트
- `TRACEABILITY.md`: 실행 로그, 재현, 이상 행동 감지의 최소 기준
- `PR_GUIDE.md`: `.github/PULL_REQUEST_TEMPLATE.md` 기반 PR 작성 규칙

## MVP 범위

포함한다.

- Jira Task 기준 컨텍스트 수집
- 설계 문서 기반 컨텍스트 요약
- 코드베이스, 테스트, API, DB, 보안, 팀 규칙의 컨텍스트 소스 정의
- Context Pack 우선순위와 크기 제한 정의
- Agent 실행 정책과 위험 작업 승인 기준 정의
- Tool Registry와 역할별 도구 권한 정의
- PR 전 검증 게이트 정의
- 작업 결과에 주입 컨텍스트를 추적하는 기준 정의
- PR 생성 시 템플릿 준수 기준 정의

제외한다.

- 런타임 게이트웨이 구현
- 프롬프트 오케스트레이션 서버
- 자동 정책 엔진
- 중앙 집중식 모니터링 파이프라인
- 운영 API 변경

## 기본 작업 순서

각 Jira 작업은 다음 순서로 진행한다.

1. Jira 키와 작업 범위를 확인한다.
2. 현재 브랜치와 작업 트리 상태를 확인한다.
3. `CONTEXT_PACK.md` 기준으로 필요한 컨텍스트를 수집한다.
4. 설계 맥락이 필요하면 `DESIGN_CONTEXT.md`를 확인한다.
5. `EXECUTION_POLICY.md`와 `TOOL_REGISTRY.md` 기준으로 허용된 작업인지 확인한다.
6. 파일 수정 전에 변경 의도를 설명한다.
7. 가장 작은 단위로 유효한 변경을 만든다.
8. `VERIFICATION_GATE.md` 기준으로 검증을 실행하거나 생략 사유를 남긴다.
9. 변경 파일, 검증 결과, 사용한 컨텍스트를 요약한다.
10. 사용자 승인 전에는 커밋, push, PR을 진행하지 않는다.

## 승인 게이트

사용자 승인이 필요하다.

- 커밋 생성
- 브랜치 push
- Pull Request 생성
- destructive Git 작업
- 요청 범위 밖 사용자 파일 삭제
- 의존성 설치
- 네트워크 접근
- 운영 또는 클라우드 리소스 변경

## 보안 원칙

- 시크릿, 토큰, 개인 인증 정보는 프롬프트, 커밋, Jira 댓글, PR 본문에 포함하지 않는다.
- 사용자가 요청하지 않은 ignored 로컬 문서는 읽지 않는다.
- 기억에 의존하지 않고 저장소의 실제 파일을 우선 확인한다.
- 코드 검색은 `rg` 또는 `rg --files`를 우선 사용한다.

## 리뷰 게이트

사용자에게 승인 요청 전 다음을 제공한다.

- 무엇을 변경했는지
- 왜 변경했는지
- 어떤 컨텍스트를 사용했는지
- 어떤 검증을 했는지
- 남은 위험이나 확인할 점이 있는지

요청보다 더 단순하거나 나은 방향이 있으면, 에이전트는 무조건 동의하지 않고 그 방향을 설명한다.
